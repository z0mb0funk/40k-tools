# MFM Tool — Codebase Guide

Read this to understand the whole project before changing it. It is the source
of truth for architecture, conventions, and the regeneration workflow. Keep it
updated when structure or invariants change.

## What this is

A personal Warhammer 40,000 tool with two halves that share one JSON schema:

1. **Points comparison** — ingest MFM points from the previous edition (PDF,
   v4.3) and the current edition (live site, v1.0), then diff them per faction
   and across the board.
2. **List builder** — a browser app to build an army for any faction with
   correct iterative pricing, wargear, character attachment, enhancements /
   upgrades, and detachment-points budgeting.

Stack: **Python 3** (pipeline) + **vanilla JS/HTML/CSS** (web app, no framework,
no build step). Python deps: `beautifulsoup4`, `pdfplumber`, `requests`.

## Mental model

```
previous_MFM.pdf ──parse_pdf──┐
                              ├─► Snapshot JSON (old.json / new.json)
live MFM site  ──parse_html──┘            │
                                          ├─diff──► diff.json ─┐
                                          │                    ├─build-app─► web/data.js
                                          └────────── new.json ┘                │
                                                                     web/index.html (file://)
```

Going forward the PDF is a **one-time baseline**; future updates are web-vs-web
(re-scrape `new.json`, diff against the previous snapshot). See README "Going
forward (web-only)".

## Data schema (models.py) — the contract

- `Snapshot{ source:"old"|"new", version, factions:{slug:Faction}, unmapped[] }`
- `Faction{ slug, name, units[Unit], enhancements[Enhancement], detachments[Detachment] }`
- `Unit{ name, name_key, cost_blocks[CostBlock], wargear[WargearOption],
        leader_targets[str], is_character:bool, attach_type:"leader"|"support"|"leader/support"|None }`
- `CostBlock{ threshold:str, tiers[CostTier] }`  threshold is "default" or an
  iterative label like "1st to 2nd", "3rd+", "1st", "2nd+".
- `CostTier{ models:int, points:int }`
- `Enhancement{ name, name_key, points, detachment, is_upgrade:bool }`
- `WargearOption{ name, points }`   (per-item cost; e.g. "per Heavy mining laser")
- `Detachment{ name, dp:int|None }` (dp only exists in the new edition)

Everything serializes to plain dicts; `Snapshot.from_dict/to_dict` round-trips JSON.

## Domain concepts (must-know)

- **name_key**: lowercased, ASCII-folded, punctuation-stripped key used to match
  units/enhancements across editions and to look up units in the builder.
- **Iterative / requisition cost**: some units cost more for repeated copies.
  The threshold scheme VARIES by faction: GSC uses `1st to 2nd` then `3rd+`;
  Chaos Knights use `1st` then `2nd+`. Always resolve the block for the Nth copy
  via `normalize.threshold_range(threshold) -> (lo, hi)` (JS mirror:
  `thresholdRange`). Never hardcode "3rd".
- **base tier**: the lowest model-count tier. All cross-edition deltas compare
  the base tier. Multi-tier-only changes show Δ0 at base but appear in the tier
  columns.
- **Enhancements vs Upgrades**: the MFM tags upgrades with a `(Upgrade)` suffix
  (`normalize.is_upgrade_name`). Enhancements go on **characters**; upgrades go
  on **non-character units**. Army rule enforced in builder: max **4**
  enhancement/upgrade selections total; an upgrade may be applied to up to **3**
  units (counts as one selection); each enhancement is unique. Upgrade points
  are paid **per instance**.
- **Leader vs Support**: both are characters that attach to a target unit list,
  but are different abilities. `attach_type` distinguishes them; the builder
  badges Leader (gold) / Support (blue) / Ldr/Sup (purple).
- **Detachment Points (DP)**: new-edition detachments cost DP; budget default 3.
  The builder allows multiple detachments up to the budget and pools their
  enhancements/upgrades.
- **Wargear**: NEW in 11th (was free in 10th), so the old PDF has none. In the
  comparison wargear is reported as new costs, never as a delta.
- **Sign convention** (diff/report/app): negative = cheaper (buff),
  positive = pricier (nerf). The overview headline metric is **avg_pct**
  (mean per-unit % change), not the raw net.

## Pipeline specifics & gotchas

- **parse_pdf.py**: two/three-column "newspaper" layout. Columns detected by a
  coverage histogram (`_column_bands`) on a fixed ~150px grid. Faction comes
  from the large-font running header (`_page_header`, size>=20). Continuation
  pages (no header) carry the current faction forward. **Legends/Forge World is
  intentionally skipped** (headers without a `CODEX:`/`INDEX:` prefix).
  `CODEX SUPPLEMENT:` pages are read as the SM chapters (black/blood/dark).
- **parse_html.py**: live pages are React streaming SSR — points sit in
  `<template id="P:x">` placeholders resolved by `$RS("S:x","P:x")`.
  `_resolve_streaming()` fills them in before parsing; a browser-saved page
  needs no resolution. Units live under `<h3>UNITS</h3>`, detachments under
  `<h3>DETACHMENTS</h3>`; unit/detachment cards share class
  `flex flex-col space-y-1 m-1`.
- **parse_int is comma-tolerant** ("2,100" -> 2100). Big models broke the diff
  before this; keep it.
- **diff.py**: matches factions by slug, units by name_key, tiers by model
  count. SM chapters fall back to vanilla `space-marines` via `FACTION_PARENTS`
  for shared units (and enhancements) without surfacing ~90 vanilla units as
  "removed". Enhancement renames/hyphenation surface as add+remove (known limit).

## Web app specifics

- No server needed: `index.html` loads `data.js` then `app.js` via `<script>`
  tags (works on `file://`; `fetch` would not). Regenerate `data.js` with
  `build-app` whenever `diff.json`/`new.json` change.
- `app.js` is one IIFE. State for the builder is `const builder = {slug,
  detachments[], dpBudget, battleSize, instances[], seq}`. An `instance` =
  `{id, key, name, isCharacter, size, wargear:{name:qty}, enhancement, upgrade,
  attachedTo}`. Top-level instances render as cards; attached characters render
  nested (filtered by `attachedTo`).
- Drag-and-drop uses a module-scoped `dragPayload` ({t:'unit'|'enh'|'reorder'}).
  Reorder uses a drag handle; characters attach by dropping on a valid unit
  card; enhancements/upgrades drop on a character/unit slot.
- Saved lists persist to `localStorage` ("mfm_saved_lists"); `Copy` exports text.
- Key render fns: `renderOverview`, `renderFaction`, `renderList`,
  `renderAvailable`, `renderEnhancements`, `renderDetachments`. Cost logic:
  `copyCost` (uses `blockForOrdinal`+`thresholdRange`), `instanceCost`,
  `selectionState`/`tryAssign` (4/3 caps), `dpUsed`.

## Regeneration workflow (run from `mfm-tool/`, with `PYTHONPATH=src`)

```bash
# old baseline (one-time / when the PDF changes)
python -m mfm.cli ingest-old --pdf /path/to/previous_MFM.pdf --out data/old.json

# new edition (all factions; GSC fixture optional for the saved page)
python -m mfm.cli ingest-new --all --cache data/raw \
  --fixture "genestealer-cults=../<saved page>.html" --out data/new.json

# comparison + report
python -m mfm.cli compare --old data/old.json --new data/new.json \
  --out data/report.md --json data/diff.json

# embed data for the web app, then open web/index.html
python -m mfm.cli build-app --diff data/diff.json --new data/new.json --out web/data.js
```

After ANY parser/model change, re-run `ingest-new` (and `ingest-old` if the PDF
parser changed) so the JSON reflects new fields, then `compare` then `build-app`.

## Verification (always do before declaring done)

```bash
python -c "import compileall; compileall.compile_dir('src', quiet=1)"   # python
node --check web/app.js                                                  # js syntax
```
For builder logic, a jsdom smoke test is the reliable check (no browser). Pattern
used during development: `new JSDOM(index.html, {url:'https://localhost/'})`,
`window.eval(data.js)`, `window.eval(app.js)`, then dispatch events and assert on
rendered DOM / totals. Install a Node-18-compatible jsdom (e.g. `jsdom@24`).

## Conventions

- Keep `models.py` the single schema; add fields with `from_dict` defaults so old
  JSON still loads.
- Mirror any Python cost/threshold logic that the builder needs in `app.js`
  (e.g. `threshold_range` <-> `thresholdRange`).
- Don't hand-edit `data/*.json` or `web/data.js`; regenerate them.
- Comments explain WHY (layout quirks, rules), not WHAT.

## Change recipes (where to touch)

- New per-unit datum from the site: add field to `Unit` (models.py) ->
  parse in `parse_html.py` -> surface in `diff.py`/`report.py` and/or `app.js`
  -> re-ingest + build-app.
- New comparison metric: compute in `diff.py` summary -> render in `report.py`
  and `app.js` (`OV_COLS` / `renderFaction`).
- New builder behavior: edit the relevant `render*`/cost fn in `app.js`; update
  `snapshot()`/`loadList()` if it adds saved state; build-app to refresh data.
